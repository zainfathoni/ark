Core Loop

How Symphony Works

Symphony is a long-running daemon. Every polling.interval_ms (default 30s), it runs this exact sequence:

Step 1
Reconcile
Check running agents against tracker state. Kill stale runs.
→
Step 2
Validate
Pre-flight config check. If broken, skip dispatch but keep reconciling.
→
Step 3
Fetch
Query tracker for active issues. Sort by priority → created_at → identifier.
→
Step 4
Dispatch
Launch agents in isolated workspaces until concurrency slots exhausted.
Architecture

System Components

Workflow Loader

Reads WORKFLOW.md from your repo. Parses YAML front matter (config) and Markdown body (prompt template). Watches for changes and hot-reloads without restart.

Config Layer

Typed getters for all workflow settings. Handles defaults, $ENV_VAR expansion, path normalization. Validates before every dispatch tick.

Issue Tracker Client

Currently Linear-only. Fetches candidates in active states, refreshes states for reconciliation, fetches terminal issues for cleanup. Normalizes everything into a stable domain model.

Orchestrator

The brain. Owns the poll tick, all in-memory state, dispatch/retry/stop decisions, and token accounting. Single authority — no external DB required.

Workspace Manager

Maps issue identifiers to filesystem paths. Creates directories, runs lifecycle hooks (after_create, before_run, etc.), cleans up terminal workspaces. Safety: all paths validated inside workspace root.

Agent Runner

Creates workspace → builds prompt → launches codex app-server via JSON-RPC over stdio → streams events back to orchestrator. Supports multi-turn sessions on the same thread.

Repository Contract

WORKFLOW.md — The Single Source of Truth

This is what makes Symphony different. Your agent's behavior is version-controlled with your code, not in some external config.

WORKFLOW.md — Front Matter (Config)
--- tracker: kind: linear project_slug: "my-project-abc" active_states: ["Todo", "In Progress"] terminal_states: ["Done", "Closed", "Cancelled"] polling: interval_ms: 30000 workspace: root: ~/code/workspaces hooks: after_create: | git clone --depth 1 git@github.com:org/repo.git . npm install before_run: | git checkout main && git pull after_run: | echo "Run complete for $ISSUE_ID" agent: max_concurrent_agents: 5 max_turns: 20 max_retry_backoff_ms: 300000 codex: command: codex app-server approval_policy: never thread_sandbox: workspace-write turn_timeout_ms: 3600000 stall_timeout_ms: 300000 ---
WORKFLOW.md — Prompt Template (Markdown Body)
You are working on Linear issue {{ issue.identifier }}. ## Task **Title:** {{ issue.title }} **Description:** {{ issue.description }} ## Instructions 1. Read the issue carefully 2. Create a feature branch 3. Implement the changes 4. Write tests 5. Open a PR with a clear description 6. Link the PR to the Linear issue 7. Move the issue to "Human Review" {% if attempt %} ## Retry Context This is attempt #{{ attempt }}. Check what was done previously and continue. {% endif %} ## Rules - Follow existing code conventions - Run tests before committing - Keep PRs focused and small
Key insight: The prompt template uses Liquid-compatible syntax with strict variable checking. Unknown variables or filters fail the render — no silent fallbacks. The issue object includes all normalized fields (labels, blockers, priority, etc.) so you can build sophisticated per-repo workflows.

Resilience

Retry & Reconciliation

Retry Logic

  • Normal exit: 1-second continuation retry. Re-checks if issue is still active, then re-dispatches.
  • Abnormal exit: Exponential backoff: min(10s × 2^(attempt-1), max_retry_backoff_ms)
  • Before re-dispatch: re-fetches active candidates, checks eligibility, verifies concurrency slots
  • If issue no longer eligible: releases claim, no retry

Reconciliation (every tick)

  • Part A — Stall detection: If no Codex events for stall_timeout_ms, kill the worker and queue retry
  • Part B — Tracker state refresh: Fetch current states for all running issues:
    • Terminal state → kill worker + clean workspace
    • Still active → update in-memory snapshot
    • Neither active nor terminal → kill worker, keep workspace
  • If state refresh fails: keep workers running, try next tick
Agent Protocol

Codex App-Server Integration

Symphony communicates with Codex via JSON-RPC over stdio. The handshake is strict — four messages in exact order:

→ {"id":1, "method":"initialize", "params":{"clientInfo":{"name":"symphony","version":"1.0"}, "capabilities":{}}} ← {"id":1, "result":{...}} → {"method":"initialized", "params":{}} → {"id":2, "method":"thread/start", "params":{"approvalPolicy":"never", "sandbox":"workspace-write", "cwd":"/workspaces/ABC-123"}} ← {"id":2, "result":{"thread":{"id":"thread_abc123"}}} → {"id":3, "method":"turn/start", "params":{"threadId":"thread_abc123", "input":[{"type":"text","text":"<rendered prompt>"}], "cwd":"/workspaces/ABC-123", "title":"ABC-123: Fix login bug"}} ← streaming events...

Turn Completion

turn/completed → success
turn/failed → failure
turn/cancelled → failure
Timeout or process exit → failure

Approval Policy

Auto-approve command + file changes. User-input-required = hard failure. Unsupported tool calls → return error + continue session.

linear_graphql Tool

Optional client-side tool that lets Codex query/mutate Linear using Symphony's auth. Single GraphQL operation per call.

Safety

Safety Invariants

The spec calls these "the most important portability constraint."

Invariant 1

Agent only runs in the per-issue workspace path. Validated: cwd == workspace_path before launch.

Invariant 2

Workspace path must stay inside workspace root. Both normalized to absolute. Prefix check enforced.

Invariant 3

Workspace key is sanitized: only [A-Za-z0-9._-] allowed. Everything else replaced with _.

Elixir Implementation

Reference Implementation

Why Elixir?

Erlang/BEAM/OTP excels at supervising long-running processes. Hot code reloading means you can update Symphony without stopping running agents. The implementation includes:

  • Phoenix LiveView dashboard at / (opt-in via --port)
  • JSON API at /api/v1/state, /api/v1/<identifier>, /api/v1/refresh
  • linear_graphql client-side tool for in-session Linear mutations
  • Skills bundled: commit, push, pull, land, linear
  • Embedded PostgreSQL not required — in-memory state, file-system recovery
Setup note: Uses mise for Elixir/Erlang version management. The Elixir impl expects custom Linear workflow states: "Rework", "Human Review", and "Merging" — configure in Linear Team Settings → Workflow. Ryan Carson reported 2-3 days to set up for his codebase.

Takeaways

What's Worth Stealing

🎯 Adopt

  • In-repo WORKFLOW.md: Let each repo define its own agent behavior. Prompt + config + hooks, version-controlled.
  • Per-issue workspaces: Git worktrees per issue. No cross-contamination between concurrent agents.
  • Reconciliation loop: Every poll tick, check running agents against tracker state. Auto-kill stale/terminal runs.
  • Lifecycle hooks: after_create / before_run / after_run give repos full control over workspace setup.

🤔 Skip (for now)

  • Linear dependency: We use GitHub Issues + Fizzy. The spec is Linear-only for now.
  • Codex-only agent protocol: We use multiple agents. Symphony's JSON-RPC protocol is Codex-specific.
  • Elixir runtime: Adding Elixir to our stack is overhead. The concepts port to our Node.js setup.
  • Full service daemon: We already have OpenClaw as our daemon. Don't need a second one.
